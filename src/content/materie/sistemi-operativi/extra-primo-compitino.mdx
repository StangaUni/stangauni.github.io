## 1. Sistema Operativo — Concetti Base

> Rif. → `Riassunti/01. Introduzione ai sistemi operativi`

### Definizione e ruolo

Un **Sistema Operativo** è un insieme di utilità che:
1. Offre un'**astrazione** più semplice e potente della macchina fisica (macchina virtuale)
2. **Gestisce ottimamente le risorse** fisiche e logiche (CPU, memoria, I/O)

### Architetture del S/O

| Architettura | Descrizione | Pro | Contro |
|---|---|---|---|
| **Monolitica** | Un unico file oggetto; ogni procedura può chiamarne un'altra | Semplice, prestazioni elevate | Nessun information hiding |
| **A strati** | Livelli gerarchici; ogni livello usa solo quello sotto | Modularità | Chiamate più costose |
| **Microkernel** | Nucleo minimo (processi + IPC); resto in processi utente | Flessibilità, affidabilità | Prestazioni scadenti |
| **Macchina virtuale** | Virtualizzazione hardware sotto vari S/O | Isolamento completo | Complessità |

### System call

- I servizi del S/O sono invocati tramite **procedure di libreria** che nascondono le system call
- La prima istruzione di ogni system call è una **trap**: forza il passaggio in modo privilegiato
- **Interruzioni:** asincrone (evento esterno). **System call:** sincrone (richiesta esplicita del processo)

**Sequenza di esecuzione:**
1. Il programma chiama la procedura di libreria
2. Parametri sullo stack; ID della chiamata in un registro noto al S/O
3. `trap` → modo privilegiato
4. Il S/O identifica ed esegue la chiamata; ritorna al chiamante

## 2. Processi e Thread

> Rif. → `Riassunti/02. Processi e thread`

### Spazio di indirizzamento di un processo (UNIX)

```
↑ alto
┌────────────────┐
│   Stack        │  pila dei contesti; cresce verso il basso; R/W
├────────────────┤
│   BSS          │  dati non inizializzati; dimensione variabile; R/W
├────────────────┤
│   Data segment │  dati inizializzati (malloc); dimensione variabile
├────────────────┤
│   Text segment │  codice macchina; dimensione fissa; R (condivisibile)
└────────────────┘
↓ basso
```

### PCB — Process Control Block

| Campo | Contenuto |
|---|---|
| Identificatore (PID) | Identificatore unico del processo |
| Contesto di esecuzione | Registri CPU da ripristinare dopo sospensione |
| Stato | Pronto / Esecuzione / Attesa |
| Priorità | Iniziale e corrente |
| Diritti di accesso | Privilegi e risorse autorizzate |
| Discendenza familiare | Puntatori a genitore e figli |
| Risorse assegnate | Lista delle risorse in uso |

### Diagramma degli stati del processo

```
             Crea()
               │
               ▼
           [PRONTO] ◄──────────── Notifica()
               │                      ▲
    Avvia()    │                      │
               ▼                      │
         [ESECUZIONE] ──Attendi()──► [ATTESA]
               │
    Prerilascio│→ [PRONTO]
               │
           Termina()
               │
               ▼
         [TERMINATO]
```

### fork()

```c
pid_t pid = fork();
// Padre: pid = PID del figlio (> 0)
// Figlio: pid = 0
// Errore: pid = -1
```

**Quanti processi crea N fork() in sequenza?** $2^N$ processi totali.

**Esempio con rami:**
```c
if (fork() == 0) { fork(); }  // Processi: padre, figlio1, figlio1's-figlio = 3
```

### Thread

**Condivisi tra thread dello stesso processo:**
- Spazio di indirizzamento (codice, dati globali, heap)
- File aperti, variabili globali

**Privato per ogni thread:**
- Stack (record di attivazione)
- Registri CPU (contesto)
- Stato (pronto / in esecuzione / in attesa)

| | Processo | Thread |
|---|---|---|
| Spazio di indirizzamento | Privato | Condiviso |
| Creazione | Costosa (fork) | Economica |
| Context switch | Pesante | Leggero |
| Comunicazione | IPC | Variabili condivise |

### Thread POSIX (pthread)

```c
pthread_create(&tid, NULL, funzione, argomento);  // crea thread
pthread_join(tid, NULL);                           // attende terminazione
pthread_exit(NULL);                                // thread termina sé stesso
pthread_mutex_lock(&m);                            // P (acquisisce mutex)
pthread_mutex_unlock(&m);                          // V (rilascia mutex)
```

## 3. Scheduling

> Rif. → `Riassunti/03. Ordinamento dei processi`

### Componenti

| Componente | Ruolo |
|---|---|
| **Scheduler** | Decide quale processo eseguire |
| **Dispatcher** | Attua la scelta: salva contesto uscente, installa entrante (context switch) |

### Metriche

| Metrica | Definizione |
|---|---|
| **Tempo di arrivo** | Istante in cui il processo entra nel sistema |
| **Tempo di attesa** | Permanenza totale in stato PRONTO |
| **Tempo di turn-around** | Completamento − Arrivo |
| **Tempo di risposta** | Prima risposta − Arrivo |
| **Throughput** | Processi completati per unità di tempo |

> Tempo di turn-around = Tempo di attesa + Tempo di esecuzione (+ eventuali prerilasci)

### Algoritmi per sistemi batch

#### FCFS — First Come First Served

Senza prerilascio; ordine di arrivo.

**Esempio:** P1(arr=0, dur=24), P2(arr=0, dur=3), P3(arr=0, dur=3)
```
Gantt: [P1: 0–24][P2: 24–27][P3: 27–30]
Attesa: P1=0, P2=24, P3=27  →  Media = 17
```

#### SJF — Shortest Job First

Senza prerilascio; esegue prima il job più breve.

#### SRTN — Shortest Remaining Time Next

SJF **con prerilascio**: se il nuovo processo è più breve → prerilascio immediato.

### Algoritmi per sistemi interattivi

#### Round Robin (RR)

Con prerilascio, coda circolare; ogni processo esegue al più per un **quanto q**.

**Esempio RR con q=2:**
P1(arr=0, dur=5), P2(arr=0, dur=3), P3(arr=0, dur=4)
```
Gantt: [P1:0-2][P2:2-4][P3:4-6][P1:6-8][P2:8-9][P3:9-11][P1:11-12]
```

### Come costruire il diagramma di Gantt

1. Disegna l'asse del tempo da 0 a T
2. Ad ogni istante scegli il processo da eseguire secondo l'algoritmo
3. Segna quando un processo arriva, viene eseguito, viene prerilasciato o termina
4. Calcola le metriche sommando i tempi di attesa e turn-around

### Inversione di priorità

**Scenario:** L (bassa), M (media), H (alta priorità); L e H condividono risorsa R.

1. L acquisisce R
2. H vuole R → **H deve attendere L**
3. M preleva la CPU a L
4. H attende indefinitamente

**Caso reale:** Mars Pathfinder (1997) — reset di sistema in VxWorks.

**BPI:** L eredita la priorità di H → completamento senza interruzioni. Non previene deadlock.

**ICP:** ogni risorsa ha priority ceiling = max priorità tra i processi che possono usarla. Un processo ad alta priorità è bloccato **al più una volta**. **Previene il deadlock.**

## 4. Sincronizzazione tra Processi

> Rif. → `Riassunti/04. Sincronizzazione tra processi`

### Il problema della race condition

**Sezione critica:** porzione di codice che accede a una risorsa condivisa.

### Condizioni di ammissibilità

1. **Accesso esclusivo** — al più un processo alla volta nella sezione critica
2. **Attesa finita** — un processo in attesa entra in tempo finito
3. **Nessuna assunzione sull'ambiente**
4. **Indipendenza dall'esterno**

### Semafori (Dijkstra, 1965)

**P (down, wait):** decrementa il contatore; se < 0 → sospende il processo.
**V (up, signal):** incrementa il contatore; se ≤ 0 → sveglia il primo processo in coda.

**Interpretazione del valore:**
- `valore > 0`: N unità ancora disponibili
- `valore = 0`: risorsa esaurita, nessuno in attesa
- `valore < 0`: |valore| processi in coda d'attesa

### Perché sleep/wakeup non bastano

Se la `wakeup` arriva prima della `sleep`, viene persa → blocco permanente.

### Produttore-Consumatore con semafori

```c
semaforo mutex     = 1;  // mutua esclusione sul buffer
semaforo non_pieno = N;  // slot liberi nel buffer
semaforo non_vuoto = 0;  // elementi presenti nel buffer

void produttore() {
    while (1) {
        prod = produci();
        P(&non_pieno);   // attende uno slot libero
        P(&mutex);
        inserisci(prod);
        V(&mutex);
        V(&non_vuoto);   // segnala elemento disponibile
    }
}
void consumatore() {
    while (1) {
        P(&non_vuoto);   // attende un elemento
        P(&mutex);
        prod = preleva();
        V(&mutex);
        V(&non_pieno);   // segnala slot liberato
        consuma(prod);
    }
}
```

> **CRITICO:** l'ordine delle P non può essere invertito. Se il produttore fa `P(&mutex)` prima di `P(&non_pieno)` e il buffer è pieno → **deadlock**.

### Monitor (Hoare, 1974)

- Solo un processo alla volta è attivo nel monitor
- `wait(cond)`: sospende il chiamante e rilascia il monitor
- `signal(cond)`: risveglia il primo processo in attesa
- `signal` **non ha memoria**: va perso se nessuno aspetta (a differenza di V sui semafori)

## 5. Deadlock

> Rif. → `Riassunti/05. Deadlock`

### 4 Condizioni di Coffman (1971)

Il deadlock si verifica se e **solo se** tutte e 4 le condizioni sono soddisfatte:

| # | Condizione |
|---|---|
| 1 | **Accesso esclusivo** |
| 2 | **Accumulo di risorse** |
| 3 | **Inibizione di prerilascio** |
| 4 | **Attesa circolare** |

### Grafo di assegnazione — come leggere il deadlock

1. Costruisci il grafo con archi R→P (assegnazione) e P→R (richiesta)
2. Cerca cicli
3. Se ci sono cicli e tutte le risorse nel ciclo hanno una sola istanza → **deadlock certo**
4. Se alcune risorse hanno più istanze → verifica se le istanze extra sono libere

### Algoritmo del Banchiere — procedura in 3 passi

1. Calcola la matrice **Necessità** = Massimo − Assegnate
2. Parte da **Lavoro** = Disponibili
3. Cerca iterativamente il processo i con `Fine[i]=false` e `Necessità[i] ≤ Lavoro`; aggiorna Lavoro con le risorse di i; segna `Fine[i]=true`; ripeti. Se tutti i Fine sono true → **sicuro**.

#### Algoritmo di richiesta delle risorse

```
I.   Se Richieste[i] > Necessità[i]  → ERRORE

II.  Se Richieste[i] > Disponibili   → Pᵢ deve attendere

III. Simula l'assegnazione:
     Disponibili  -= Richieste[i]
     Assegnate[i] += Richieste[i]
     Necessità[i] -= Richieste[i]

IV.  Esegui algoritmo di sicurezza:
     Se SICURO  → assegna definitivamente le risorse
     Se INSICURO → Pᵢ attende; annulla la simulazione
```

#### Esempio completo

**Sistema:** 4 processi (A, B, C, D), 5 tipi di risorse.
`Disponibili = [0, 1, 3, 1, 2]`

| Processo | Assegnate | Massimo | Necessità |
|---|---|---|---|
| A | 1 0 2 1 1 | 3 1 2 1 3 | **2 1 0 0 2** |
| B | 2 0 1 1 1 | 3 3 4 2 1 | **1 3 3 1 0** |
| C | 1 1 0 1 0 | 2 1 4 1 0 | **1 0 4 0 0** |
| D | 1 1 1 1 0 | 1 1 3 2 1 | **0 0 2 1 1** |

**Esecuzione** (Lavoro iniziale = `[0 1 3 1 2]`):

| Passo | Processo | Necessità ≤ Lavoro? | Nuovo Lavoro |
|---|---|---|---|
| 1 | **D** | `[0 0 2 1 1]` ≤ `[0 1 3 1 2]` ✓ | `[1 2 4 2 2]` |
| 2 | **C** | `[1 0 4 0 0]` ≤ `[1 2 4 2 2]` ✓ | `[2 3 4 3 2]` |
| 3 | **A** | `[2 1 0 0 2]` ≤ `[2 3 4 3 2]` ✓ | `[3 3 6 4 3]` |
| 4 | **B** | `[1 3 3 1 0]` ≤ `[3 3 6 4 3]` ✓ | — |

**Sequenza sicura: D → C → A → B** → stato sicuro.

## 6. Domande Vero/Falso Frequenti agli Esami

| Affermazione | Risposta |
|---|---|
| Un ciclo nel grafo di assegnazione implica sempre deadlock | **FALSO** — solo se le risorse hanno molteplicità 1 |
| Uno stato non sicuro implica necessariamente deadlock | **FALSO** — solo che non si può garantire l'assenza |
| BPI previene il deadlock | **FALSO** — previene solo l'inversione di priorità |
| ICP previene il deadlock | **VERO** |
| SRTN e SJF producono lo stesso risultato in assenza di arrivi dopo t=0 | **VERO** |
| Round Robin con quanto infinito equivale a FCFS | **VERO** |
| Il valore negativo di un semaforo indica il numero di processi in attesa | **VERO** — `|valore|` processi in coda |
| `signal` su una variabile di condizione del monitor va perso se nessuno aspetta | **VERO** — a differenza di V su semaforo |
| I thread di uno stesso processo condividono la heap | **VERO** — condividono heap, codice e file; ogni thread ha il suo stack |
| `fork()` restituisce 0 al padre e il PID del figlio al figlio | **FALSO** — è il contrario: 0 al figlio, PID al padre |

## 7. Riepilogo Metriche di Scheduling

$$\text{Attesa}_i = \text{Inizio esecuzione}_i - \text{Arrivo}_i \quad \text{(senza prerilascio)}$$

$$\text{Turn-around}_i = \text{Fine}_i - \text{Arrivo}_i$$

$$\text{Attesa media} = \frac{\sum_i \text{Attesa}_i}{N}$$

**Con prerilascio**, il tempo di attesa è la somma di tutti i periodi in stato PRONTO.

## 8. Schemi Rapidi per gli Esercizi

### Scheduling con Gantt — procedura

1. Metti tutti i processi in una lista con arrivo e durata
2. All'istante t: scegli il processo da eseguire secondo l'algoritmo
3. Avanza il tempo: di 1 unità (per algoritmi con prerilascio) o fino alla fine del processo (FCFS/SJF)
4. All'arrivo di nuovi processi (SRTN/RR): reinserisci nella coda e rivaluta
5. Al termine: attesa = Σ(periodi in stato PRONTO); turn-around = Fine − Arrivo

### Come tracciare l'esecuzione di codice con semafori

1. Disegna una colonna per ogni processo; una riga per ogni istante/passo
2. Per ogni `P(s)`: decrementa s; se s < 0 → il processo si blocca
3. Per ogni `V(s)`: incrementa s; se s ≤ 0 → sveglia il primo in coda
4. Marca il processo sbloccato come pronto; eseguilo al prossimo passo disponibile

### Algoritmo del Banchiere — procedura passo-passo

1. Calcola **Necessità** = Massimo − Assegnate per ogni processo
2. Imposta **Lavoro** = Disponibili; **Fine[i]** = false per tutti
3. Cerca processo i con Fine[i]=false e Necessità[i] ≤ Lavoro → aggiorna Lavoro += Assegnate[i]; Fine[i] = true
4. Ripeti fino a che non trovi più processi → se tutti Fine[i]=true → **SICURO**

### Riepilogo meccanismi di sincronizzazione

| Meccanismo | Memoria condivisa | Busy wait | Ammissibile |
|---|---|---|---|
| TSL / spin-lock | Sì | Sì | Sì |
| Peterson | Sì | Sì | Sì (2 processi) |
| Semaforo | Sì | No | Sì |
| Monitor | Sì | No | Sì (a livello linguaggio) |
| Message passing | No | No | Sì |
