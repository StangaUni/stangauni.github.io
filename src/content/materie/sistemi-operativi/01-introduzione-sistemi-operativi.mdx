## Definizione

Un **Sistema Operativo** è un insieme di utilità progettate per:
1. **Offrire un'astrazione** più semplice e potente della macchina fisica (macchina virtuale)
2. **Gestire ottimamente le risorse** fisiche e logiche (CPU, memoria, I/O)

## Nozione di processo

Un **processo** è un programma in esecuzione. Corrisponde a:
- L'insieme ordinato degli **stati** assunti nel corso dell'esecuzione (*automa a stati*)
- L'insieme ordinato delle **azioni** effettuate nel corso dell'esecuzione (*attore*)

Ogni processo ha uno **spazio di indirizzamento logico** (memoria virtuale) contenente programma, dati e aree di lavoro.

### PCB — Process Control Block

Ogni processo è rappresentato da un descrittore (**PCB**) nella tabella dei processi:

| Campo | Contenuto |
|---|---|
| Identificatore | PID del processo |
| Contesto di esecuzione | Registri CPU (da ripristinare dopo sospensione) |
| Stato di avanzamento | Pronto / Esecuzione / Attesa / ... |
| Priorità | Iniziale e corrente |
| Diritti di accesso | Privilegi e risorse autorizzate |
| Discendenza familiare | Puntatori a genitore e figli |
| Risorse assegnate | Lista delle risorse in uso |

## Tipi di processo e scambio

- **CPU-bound:** lunghe fasi di elaborazione CPU
- **I/O-bound:** brevi fasi CPU alternate a lunghe attese I/O (FCFS penalizza questi)

**Scambio cooperativo:** il processo decide da solo quando cedere il controllo (es. Windows 3.1) — nessuna garanzia di fairness.

**Scambio a prerilascio:** il S/O forza il cambio:
- Per priorità (*priority-based*, sistemi real-time)
- All'esaurimento del quanto (*time-sharing*, Unix/Windows NT)

## Risorse

**Risorsa** = qualsiasi elemento necessario alla creazione, esecuzione e avanzamento di processi.

| Proprietà | Tipi |
|---|---|
| Durata | Durevole (CPU) / Consumabile (segnale) |
| Accesso | Individuale / Molteplice |
| Divisibilità | Divisibile (tollera alternanza) / Indivisibile |

- **CPU:** risorsa indispensabile; a livello logico → macchina virtuale
- **Memoria:** scrittura → individuale; lettura → multipla; virtualizzabile con disco
- **I/O:** riutilizzabili, non prerilasciabili, accesso individuale

## Architettura hardware rilevante

### Gerarchia di memoria

| Livello | Accesso | Capacità |
|---|---|---|
| Registri CPU | ~1 ciclo | 32–64 registri |
| Cache L1 | ~2 cicli | Decine di KB |
| Cache L2 | ~10 cicli | Centinaia di KB – pochi MB |
| RAM | ~100 cicli | 16 MB – 16 GB |
| Disco magnetico | ~10 ms | 1 TB+ |

Cache: suddivisa in **line** da 64 B; write-through o copy-back. L2: condivisa (Intel) o per core (AMD).

**SSD** vs **HDD:** SSD fino a 200× più veloce, silenzioso, ~2 W; HDD più capiente ed economico. Soluzioni ibride: S/O su SSD, dati su HDD.

### Dischi magnetici

- **Tracce:** anelli concentrici sul piatto; suddivise in **settori**
- **Cilindro:** insieme delle tracce alla stessa distanza radiale su tutti i piatti

### Rilocazione e MMU

**Problema:** quando si compila un programma non si sa dove verrà caricato in RAM.

**Soluzione:** registri **base** e **limite**:
- `indirizzo fisico = base + indirizzo logico`
- Ogni accesso verifica: `indirizzo logico ≤ limite`

La **MMU** (*Memory Management Unit*) interposta tra CPU e bus esegue questa traduzione.

**Senza interruzioni:** polling (verifica continua) — spreco di CPU.

**CMOS:** memoria volatile a batteria per ora, dispositivo di boot, configurazione BIOS.

**Plug & Play** (Intel + Microsoft): il sistema assegna centralmente livelli di interrupt e indirizzi I/O per evitare conflitti.

## Boot

1. BIOS verifica RAM e periferiche; scannerizza bus ISA e PCI; configura Plug & Play
2. Legge il settore 0 del disco di boot (MBR) in RAM e lo esegue
3. L'MBR individua la partizione attiva e carica il boot loader
4. Il boot loader carica il S/O dalla partizione attiva
5. Il S/O interroga il BIOS, carica i driver, avvia il login / GUI

## Architetture del S/O

<ThemedImage
  lightSrc="/assets/sistemi-operativi/architetture_so.svg"
  darkSrc="/assets/sistemi-operativi/architetture_so_scuro.svg"
  alt="Architetture del sistema operativo"
/>

**Dibattito Torvalds vs Tanenbaum:**
- Linux = monolitico
- Minix = microkernel
- Windows / macOS = ibridi

| Architettura | Descrizione | Pro | Contro |
|---|---|---|---|
| **Monolitica** | Un unico file oggetto; ogni procedura può chiamarne un'altra | Semplice, prestazioni elevate | Nessun information hiding |
| **A strati** | Livelli gerarchici; ogni livello usa solo quello sotto | Modularità | Chiamate più costose |
| **Microkernel** | Nucleo minimo (processi + IPC); resto in processi utente | Flessibilità, affidabilità | Prestazioni scadenti |
| **Macchina virtuale** | Virtualizzazione hardware sotto vari S/O | Isolamento completo | Complessità |
